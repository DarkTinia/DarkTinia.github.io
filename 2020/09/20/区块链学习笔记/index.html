2 密码学原理

哈希函数

不可逆，抗碰撞，无法证明其安全性(collision resistance)

哈希碰撞：由于输入空间远大于输出空间(2^256)，难免会出现H(A) == H(B)的情况。

brute-force 暴力寻找碰撞。实际可行性很低

H(data | nonce) 保证足够随机。

哈希函数为puzzle friendly，即制造哈希碰撞无捷径。本质为无法人为制造哈希碰撞

difficult to solve, but easy to verity 难解易验证

比特币使用的哈希函数为sha-256(Secure Hash Algorithm - 256)

创建账户即在本地生成一公私钥对

Q:会不会本地生成的公私钥对与已存在的账户相同？

A:有可能，但概率非常非常非常非常低。

创建账户和签名时都需要好的随机源(a good source of randomness)，否则可能泄露私钥

3 数据结构

哈希指针(hash pointer)

区块链亦为链表，由哈希指针代替普通指针。哈希指针：H(data | nonce)

无环链表均可用哈希指针。

第一个区块称为创世区块(genesis block)，最后一区块的哈希指针存在系统内

merkle tree为二叉树，最底层为块身，其他层均为块头。块身保存交易信息，块头保存哈希指针。

验证交易只需验证Merkle proof，所需时间为O(logn）

验证时只需传输几个兄弟节点的哈希值即可。无法确定兄弟节点的哈希值是否正确。

Q:能否篡改交易，通过传输错误的兄弟节点的哈希来保证计算出的哈希值与上层节点保存的哈希值相同？

A:实际操作中几乎无法实现。其本质仍是人为制造哈希碰撞。

没有高效的办法来判断整颗树是否完全正确。所需时间为O(n)

已排好序的Merkle tree称为Sorted Merkle tree，比特币中并未使用该方法。

4 BTC-协议

数字货币的主要挑战就是如何规避双花攻击(double spanding attack)

比特币中每笔交易都需说明比特币的来源。用哈希指针来指向币的来源来防止双花攻击。写入区块链前会先对币的来源进行溯源来检验交易是否合法。

区块链中的双花攻击本质是利用算力来将交易回滚。原因是非最长合法链中的交易会被撤销。

块头中的信息(block header)：

版本信息version(4byte)

前一个区块头哈希值(64byte)

merkle tree的根哈希(64byte)

产生区块的时间time(4byte)

难度目标阈值 target(4byte)

随机数 nonce(4byte)

块身中的信息(block body)

交易列表

只需要取区块头的哈希即可保证信息正确。

全节点和轻节点

全节点(full node)

全节点保存区块链所有信息，需验证所有交易信息，又称为fully validating node

轻节点(light node)

轻节点无法独立验证交易的合法性。

大多数节点为轻节点。

账本的内容要有分布式共识(distributed consensus)

例如分布式哈希表(distributed hash table)

CAP Theorem

Consistency(一致性) Availability(可用性) Partition tolerance(分区容错性) 三个性质最多同时满足两个。

比特币中的共识协议(Consensus in BitCoin)

投票机制：首先需确认谁有投票权。若所有账户均有投票权则可能会招致女巫攻击(sybil attack)。比特币采取依靠算力来投票的方式(hash rate)。

任何账户可在本地生成候选区块。当尝试到一个合适的nonce时则获得记账权，可发布区块。其他账户可对其进行验证。

分叉攻击(forking attack)

若同时发布合法区块，则最终成为最长合法链中的区块会被认可，未被认可的区块称为(orphan block)。被认可的区块的发布者可得到出块奖励，即获得铸币权，且该方法为比特币发行货币的唯一方法。最初每个区块奖励50个BTC，每隔21w个区块铸币奖励减半(约4年)，最终总计会发放2100w个BTC

5 BTC-实现

区块链采用基于交易的记账方式(transaction-based ledger)

全节点需维护UTXO(Unspent Transaction Output)未花掉的交易的输出组成的集合。其中，所有输入的金额加起来需等于所有输出的金额(total inputs == total outputs)。一些交易中total inputs可能略大于total outputs，其差额作为交易费(transaction fee)付给发布该区块的账户。

挖矿时除了尝试nonce外还可修改time，还可通过修改铸币交易中的Coinbase来间接修改根哈希

比特币通过交易的输入脚本与提供币的来源的输出脚本拼接起来执行来判断交易是否合法。若脚本成功执行则交易合法。

比特币默认大部分节点为诚实节点，即诚实节点占了一半以上的算力。

诚实节点会沿最长合法发布区块，不会沿着存在非法交易的区块继续挖。

六个区块后将无法再分叉。

selfish mining 挖到区块先藏着，用于分叉攻击。基本没有成功的概率。

6 BTC-网络

The BitCoin Network simple, robust but not efficient

需维护交易池。

节点第一次听到某个消息的时候，会将其传播给所有的零度节点，同时记录一下这个消息我已经收到过。下次再收到这个消息的时候，不会再次转发给零度节点。零度节点是随机选取的，不会考虑底层的拓扑结构。

比特币的每个节点要维护一个等待上链的交易的集合(交易池)。假如一个集合的交易都是等待写入区块链里的，那么第一次听到某个交易的时候，把这个交易加入这个集合，并且转发这个交易给零度节点，以后再收到这个交易就不用转发了，这样避免交易会在网络上无线的传播下去。转发的前提是该交易是合法的。(应该是全节点而非轻节点。)

若两个冲突的交易同时被广播到网络上，则每个节点认为收到的第一个交易是合法的。

若两个交易中任意一个被发布到区块中，则另一交易被认为是不合法的，节点会将其从交易池中删除。(当然，被认为是合法的交易也会被删除，因为已经被发布了。)

比特币限制区块大小需小于1M

7 挖矿难度

挖矿本质为寻找合适的nonce使H(data | nonce) <= target，难度调整的本质是改变target的值，即调整目标空间在整个输出空间中所占的比例。

显而易见，挖矿难度与目标阈值呈反比。目标阈值越小越难找到合适的nonce

Q:为何要调整出块时间？

A:防止出块速度过快而导致频繁出现分岔。分岔多会导致算力分散，更易引发恶意节点的分岔攻击

比特币预设出块时间是10min。以太坊预设出块时间为15s，因此以太坊中会更频繁的出现分岔。但以太坊采取了新的协议(ghost)。在该协议中orphan block也会获得部分奖励并且不会被丢弃掉，这种区块称为uncle block。

比特币协议中规定每隔2016个区块要调整目标阈值，大概为两周调整一次。

具体公式为: target = target * (actual time / expected time) 其中actual time为产生2016个区块实际花费的时间，expected time指产生2016个区块所用的时间。且计算时目标阈值增大/缩小的上限为4倍

若恶意的节点不调整target则其发布的区块会被认为是非法的。

8 挖矿

全节点

  • 一直在线

  • 在本地硬盘上维护完整的区块信息

  • 在内存中维护UTXO集合，以便快速检验交易的正确性。

  • 监听比特币网络上的交易信息，验证每个交易的合法性。

  • 决定哪些交易会被打包到区块里

  • 监听别的矿工挖出来的区块，验证其合法性

  • 挖矿

    • 决定沿着哪条链挖下去

    • 当出现等长分叉时选择哪一个分叉

轻节点

  • 不是一直在线
  • 不用保存整个区块链，只需保存每个区块的块头
  • 不用保存全部交易，仅保存与自己相关的交易
  • 无法验证大多数交易的合法性，只能检验与自己相关交易的合法性。
  • 无法检测网上发布的区块的正确性
  • 可验证挖矿难度
  • 仅可检测最长链表，无法检测最长合法链。

挖矿设备越来越趋于专业化。

挖矿的另一个趋势是大型矿池的出现。

矿池

矿池将矿工组织起来，作为一个整体，矿池的架构一般是一个全节点会驱动很多矿机，一个矿池有一个矿主，叫pool manager。下面连了很多矿工，这些矿工只负责计算哈希值，全节点的其他职责都由矿主来承担。他负责监听网上的交易，把这些交易组织打包成区块，同时要看一看有没有其他的节点抢先发布区块，如果有的话看怎样进行调整。矿池的出现解决了收入不稳定的问题。

矿工按照矿池规定的通讯协议与矿主联系加入矿池。矿主把计算哈希值的任务分配给他，矿工计算完之后，把结果反馈给矿主，获得出块奖励时一起分配。

分配规则

凭借降低难度后的nonce来获得分配收益的权利。

例如，target要求前面至少有70个0才是合法的区块。降低挖矿难度之后，比如说前面只要有60个0就行了，这样挖到的叫作一个share，这个share叫做almost valid block。矿工挖到share或almost valid block之后，把它提交给矿主。矿主统计每个矿工提交了多少这样的share，等真正挖到了合法的区块之后，再将出块奖励按照每个矿工所做的工作量，即提交的share数目进行分配

Q:会不会有矿工挖到合适的nonce后不提交给矿池而是自己偷偷发布？

A:不会。因为做不到。矿工的任务由矿主分配，挖矿仅仅调nonce是不够的，还需要调整coinbase parameter。矿主将不同的coinbase parameter所对应的nonce值的范围交给不同的矿工去尝试。coinbase transaction里面有收款人的地址，这个地址填的是矿主的地址，即pool manager的地址。矿工挖到区块之后，如果他不提交给矿主自己发布出去是没有用的。里面的收款地址是矿主的。他取不出钱来。所以只要是当初按矿主分配的任务来挖矿就不可能偷取到区块奖励。

巨型矿池的存在使分叉攻击变成了可能。

除了分叉攻击外，攻击者还可进行封锁境域(boycott) 大致内容为通过在包含某账户的交易的区块处分叉来阻止该账户进行交易。

9 比特币脚本

比特币所用脚本语言被称为基于栈的脚本语言

交易结构

Code

P2PK(pay to public key)

最简单的形式。

input script //输入脚本

​ PUSHDATA(Sig)

output script //输出脚本

​ PUSHDATA(PubKey)

​ CHECKSIG

P2PKH(Pay to Public Key Hash)

input script:

​ PUSHDATA(Sig)

​ PUSHDATA(PubKey)

output script:

​ DUP //复制

​ HASH160

​ PUSHDATA(PubKeyHash)

​ EQUALVERIFY

​ CHECKSIG

P2SH(Pay to Script Hash)

最初并没有该方法。通过软分叉来更新出此方法。易于实现多重签名。

input script:

​ …

​ PUSHDATA(Sig) //提供脚本的哈希 。该脚本为readeemScript，即赎回脚本。
​ PUSHDATA(serialized redeemScript)

​ …

output script:

​ HASH160

​ PUSHDATA(redeemScriptHash)

​ EQUAL

举例。用P2SH实现P2PK

redeemScript:

​ PUSHDATA(PubKey)

​ CHECKSIG

input script:

​ PUSHDATA(Sig)

​ PUSHDATA(serialized redeemScript)

output script:

​ HASH160

​ PUSHDATA(redeemScriptHash)

​ EQUAL

Proof of Burn

outscript的一种形式。

首条语句即return。执行到return会无条件报错。即币永远无法花出去。是证明比特币可销毁的一种方式。

另一种用途是在return后写一些东西，用于永久保存。

10 分叉

deliberate fork

protocol fork 因协议造成的分叉

又分为硬分叉(hard fork)和软分叉(soft fork)

硬分叉后旧节点不认可新节点发布的区块，但新节点认可旧节点发布的区块。

硬分叉后两条链分别记账。

软分叉后新节点不认可旧节点发布的区块，但旧节点认可新节点发布的区块。此时旧节点若不及时更新则只能空耗算力

P2SH(Pay to Script Hash)就是通过软分叉更新的。
