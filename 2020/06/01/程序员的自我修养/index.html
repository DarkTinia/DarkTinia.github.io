    第三章

    程序源代码编译后机器指令常被放在代码段(.code或.text)，全局变量和局部静态变量放在数据段(.data)，未初始化的全局变量和局部静态变量放在.bss段。此外还有只读数据段(.rodata)，注释信息段(.comment)和堆栈提示段(.note.GNU-stack)等。
    为何不将未初始化的全局变量和局部静态变量也放在.data段？
    bss段只是为未初始化的全局变量和局部静态变量预留位置，它并没有内容，在文件中也不占据空间。(节约硬盘空间)
    运行时.bss段默认填充为0，因此初始化为0的全局变量与局部静态变量也被认为是未初始化的。

    在gcc中，在全局变量或函数前加attribute((section(“name”)))属性即可将相应变量或函数放到以”name”作为段名的段中
    elf文件魔数为16字节。第一字节为0x7f，对应ascii码里的del控制符。后三字节为elf这三个字母的ascii码。第五个字节用于识别elf文件类，01为32位、02为64位。第六字节规定该elf文件为大端序还是小端序。第七字节规定该elf文件的主版本号，一般为1。后九字节在elf标准中未定义，一般填0（可以藏东西？）

    符号修饰与函数签名

    编译c++代码时，gcc会使用某种名称修饰的方法使每个函数签名对应一个修饰后名称
    例如

    函数签名 修饰后名称
    int func(int) _Z4funci
    float func(float) _Z4funcf
    int C::func(int) _ZN1C4funcEi
    int C::C2::func(int) _ZN1C2C24funcEi
    int N::func(int) _ZN1N4funcEi
    int N::C::func(int) _ZN1N1C4funcEi

    不同编译器修饰方法可能不同。

    使用c++编译器可通过 extern “C” 来编译c语言代码。

    c

    由于c编译器和c++编译器对名称的修饰规则不同，且c语言不支持extern “C”语法，使用c++调用c语言库函数时会用到c++的宏__cplusplus，c++编译器在编译c++程序时会默认定义这个宏。示例如下

    c

    编译器默认函数和初始化了的全局变量为强符号，未初始化的全局变量为弱符号，也可通过gcc的”a__attribute__((weak))“来指定任何一个强符号为弱符号
    例如

    上述代码中”weak”，”weak2”为弱符号，”strong”，”main”为强符号，”ext”既非强符号也非若符号。

    第四章

    链接器将多个目标文件加工合并到输出文件时，会采用按序叠加或相似段合并，现代链接器往往使用第二种方式。
    使用这种方法的链接器一般采用两步链接的方法：
    第一步 空间与地址分配
    第二步 符号解析与重定位

    linux下32位elf可执行文件默认从地址0x08048000开始分配
    对于32位的Intel x86系列处理器来说，重定位表的结构为ELF32_Rel结构的数组。

    c

    在调用main函数前会执行.init段的代码，main结束后会执行.fini段的代码

    两个编译器编译出的目标文件能相互链链接的条件是：

    • 采用相同的目标文件格式
    • 有相同的符号修饰标准
    • 变量的内存分布方式相同
    • 函数的调用方式相同
      等等。
      其中，将符号修饰标准、变量内存布局、函数调用方式等和可执行代码二进制兼容性相关的内容称为ABI(Application Binary Interface)

    以下几个方面决定目标文件是否二进制兼容：

    • 内置类型（如int、float等）的大小和在存储器中的放置方式（大端、小端、对齐方式等）
    • 组合类型（struct、union等）的存储方式和内存分布
    • 外部符号与用户定义的符号之间的命名方式和解析方式（如func在c语言的目标文件中是否被解析成外部符号_func）
    • 函数调用方式，如参数入栈顺序，返回值如何保存
    • 堆栈分布方式，如参数和局部变量在堆栈里的位置，参数传递方式等
    • 寄存器调用约定，函数调用时哪些寄存器可修改，哪些需要保存等
      这只是一部分因素。

    而c++要做到二进制兼容比c更加困难：

    • 继承类体系的内存分布，如基类，虚基类在继承类中的位置等

    • 指向成员函数的指针的内存分布，如何通过指向成员函数的指针来调用成员函数，如何传递this指针

    • 如何调用虚函数，vtable的内容和分布形式，vtable指针在object中的位置等

    • template如何实例化

    • 外部符号的修饰

    • 全局对象的构造和析构

    • 异常的产生和捕获机制

    • 标准库的细节问题，RTTI如何实现等

    • 内嵌函数访问细节

      C++为人诟病的一大原因就是它的二进制兼容性不好，而现在C++ABI标准的统一情况仍不乐观，基本形成以微软的VISUAL C++和GNU阵营的GCC为首的两大派系，各持己见互不兼容

    第六章

    程序运行时有局部性原理。我们将程序最常用的部分驻留在内存中，而不太常用的部分放在磁盘里。此即为动态装入的基本原理

    在虚拟存储出现之前覆盖装入的应用比较广泛，覆盖装入需要手写覆盖管理器。现代操作系统往往采用页映射

    进程的建立

    • 创建一个独立的虚拟地址空间
    • 读取可执行文件头，并且建立虚拟空间与可执行文件的映射关系
    • 将cpu的指令寄存器设置成可执行文件的入口地址，启动运行

    操作系统将相同权限的段(section)合并到一起当作一个段(segment)进行映射
    ELF文件按section存储，装载时按segment划分

    ELF可执行文件有一个专门的数据结构叫程序头表用于保存segment的信息。

    同段结构一样，，程序头表也是一个结构体数组，其结构如下

    c

    其成员含义如下

    成员 含义
    p_type segment的类型
    p_offset segment在文件中的偏移
    p_vaddr segment的第一个字节在进程虚拟地址空间的起始位置
    p_paddr segment的物理装载地址
    p_filesz segment在ELF文件中所占空间的长度
    p_memsz segment在进程虚拟地址空间中所占用的长度
    p_flags segment的权限属性，如RWX
    p_align segment的对齐属性。实际对其字节等于2的p_align次方

    一个进程基本上可以分为如下几种VMA区域

    • 代码VMA。只读可执行，有映像文件
    • 数据VMA。可读可写可执行，有映像文件
    • 堆VMA。可读可写可执行，无映像文件，匿名，可向上扩展
    • 栈VMA。可读可写，不可执行，无映像文件，匿名，可向下扩展

    为了节省物理内存，有些操作系统让各个段接壤的部分共享一个物理页面，然后将该物理页面分别映射两次

    进程启动后，程序的库部分会把对战里的初始化信息中的参数信息传递给main函数，即main函数的两个参数argc和argv，这两个参数分别对应命令行参数数量和命令行参数字符串指针数组

    Linux内核装载ELF过程简介

    首先在用户层面，bash进程会调用fork()系统调用创建一个新的进程，然后新的进程调用execve()系统调用执行指定的ELF文件，原先的bash进程继续返回等待刚才启动的新进程结束，然后继续等待用户输入命令。execve()系统调用被定义在unistd.h，其原型为
    int execve(const char filename, char const argv[], char *const envp[])
    execve系统调用相应的入口是    sys_execve()，sys_execve()进行一些参数的检查和复制后调用do_execve()。do_execve()先查找被执行的文件，若找到文件则读取文件的前128字节用于判断可执行文件的格式（如ELF可执行文件，a.out,Java程序等）。判断完文件格式后调用search_binary_handle()去搜索和匹配合适的可执行文件装载处理过程。其中，load_elf_binary()**的主要步骤是:

    1. 检查ELF可执行文件的有效性。如魔数，程序头表中段(segment)的数量
    2. 寻找动态链接的”.interp”段，设置动态链接器路径
    3. 根据ELF可执行文件的程序头表的描述，对ELF文件进行映射
    4. 初始化ELF进程环境
    5. 将系统调用的返回地址修改为ELF可执行文件的入口点。静态链接的ELF可执行文件入口为ELF文件的文件头中e_entry所指地址，动态链接的ELF可执行文件程序入口点是动态链接器。
      当load_elf_binary()执行完毕，返回至do_execve()再返回至sys_execve()时，第5步已经将系统调用的返回地址改为被装在的ELF程序的入口地址。所以当sys_execve()系统调用从内核态返回到用户态时，EIP寄存器直接跳转到了ELF程序的入口地址，于是新的程序开始执行，ELF可执行文件装载完毕。

    第七章

    静态链接的缺点：浪费内存和磁盘空间，模块更新困难等。
    将链接这一过程推迟到运行时再进行即为动态链接。
    动态链接使开发过程中各个模块更加独立，耦合度更小，便于不同的开发者和开发组织之槛独立进行开发和测试。
    动态链接还有一个特点即程序运行时可动态地选择加载各种程序模块。这个优点就是后来被人们用来制作程序的插件

    动态链接的基本实现

    动态链接涉及运行时的链接及多个文件的装载，必须要有操作系统的支持，因为动态链接的情况下，进程的虚拟地址空间分布会比静态链接情况下更为复杂。
    在Linux系统中，ELF动态链接文件被称为动态共享对象，简称共享对象，一般以”.so“为扩展名。windows系统中动态链接文件被称为动态链接库，一般以”.dll“为扩展名。
    程序与libc.so之间真正的链接工作是由动态链接器完成的。
    由于动态链接的程序每次被装载时都要重新进行链接，所以动态链接会导致程序在性能上的一些损失，但延迟绑定等方法则可将性能损失尽可能的减小。据估算，动态链接与静态链接相比性能损失大约在5%以下
